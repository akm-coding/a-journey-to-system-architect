**compose-guide.md** — Docker Compose concepts:
- What Compose solves: running multiple interconnected services, consistent dev environments, one-command stack management
- Compose file anatomy: services, networks, volumes — explain YAML structure
- Service definitions: image vs build, ports, environment, volumes, depends_on
- Docker networking in Compose: automatic bridge network, service name DNS resolution (e.g., http://api:3000 resolves because Compose creates a network and registers service names)
- Health checks: healthcheck directive syntax, why "container started" != "service ready", pg_isready for PostgreSQL, redis-cli ping for Redis
- depends_on with conditions: service_started (default, weak) vs service_healthy (waits for health check)
- Volume types: named volumes (pgdata for persistence) vs bind mounts (source code for dev hot reload)
- Environment variables: environment directive vs env_file, the .env + .env.example pattern (.env is gitignored, .env.example is committed as template)
- Override files: how docker-compose.prod.yml extends docker-compose.yml, the `-f` flag for selecting configs
- Dev vs prod differences: dev uses bind mounts + dev commands, prod uses baked images + restart policies
- Use WHY/GOTCHA callouts consistent with Phase 1-2 pattern

**compose-exercise.md** — Hands-on exercise:
1. Create .env file from .env.example, filling in values
2. Start with just the db service: `docker compose up db` — verify with pg_isready
3. Add redis service, verify with redis-cli ping
4. Add api service with depends_on health checks — run `docker compose up`, observe startup order in logs
5. Add frontend service — full stack running
6. Database seeding: `docker compose exec api pnpm db:push && docker compose exec api pnpm seed`

8. Dev workflow: modify a file, observe what happens (discuss when hot reload applies vs when rebuild is needed)
9. Prod workflow: `docker compose -f docker-compose.yml -f docker-compose.prod.yml up --build`
10. Cleanup: `docker compose down`, `docker compose down -v` (explain -v removes named volumes)

**app/docker-compose.yml** — Dev configuration with 4 services:

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3

  api:
    build:
      context: .
      dockerfile: Dockerfile.api
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB}
      REDIS_URL: redis://redis:6379
      PORT: 3000
      NODE_ENV: development
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - ./src/server:/app/src/server
      - ./dist/server:/app/dist/server

  frontend:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "80:80"
    depends_on:
      - api

volumes:
  pgdata:
```
