Create `.github/workflows/ci-cd.yml` -- a HEAVILY ANNOTATED single workflow file. This is a study material, not just a config file. Every section should have comments explaining WHY, not just WHAT.

The workflow must implement (per user decisions):

**Triggers:**
- `pull_request` to `[develop, main]` -- lint, test, build images (verify they build)
- `push` to `[develop, main]` -- lint, test, build, push to ECR, deploy via SSH

**Concurrency:** Cancel in-progress runs for same branch/PR using `concurrency` group.

**Job 1: lint-and-test**
- `runs-on: ubuntu-latest`, `working-directory: ./app`
- `actions/checkout@v4`
- `pnpm/action-setup@v4` with version 10
- `actions/setup-node@v4` with node-version 20, cache 'pnpm', cache-dependency-path `pnpm-lock.yaml`
- `pnpm install --frozen-lockfile`
- `pnpm lint`, `pnpm format`, `pnpm test`
- Add block comments explaining each step's purpose

**Job 2: build-images**
- `needs: lint-and-test`
- Build frontend: `docker build -t ecommerce-frontend ./app`
- Build API: cd app, `pnpm install --frozen-lockfile`, `pnpm build:server`, `docker build -t ecommerce-api -f Dockerfile.api .`
- Comment explaining why API needs pre-build step (Dockerfile.api expects dist/server/)
- This job verifies images build. No push -- that happens in job 3.

**Job 3: push-and-deploy**
- `if: github.event_name == 'push'` (only on merge, not PR)
- `needs: build-images`
- `environment: ${{ github.ref == 'refs/heads/main' && 'production' || 'staging' }}`
- Comment block explaining GitHub Environments: same secret names, different values per env
- `aws-actions/configure-aws-credentials@v4` with secrets for access key, secret key, and vars for region
- `aws-actions/amazon-ecr-login@v2` (id: login-ecr)
- Build, dual-tag (commit SHA + latest), push both images to ECR
- `appleboy/ssh-action@v1` deploy step: authenticate Docker to ECR on EC2, `git pull` to sync compose file, `docker compose pull`, run migrations via `docker compose run --rm api pnpm db:push`, `docker compose up -d --force-recreate --remove-orphans`
- Health check: wait 15s, then retry up to 10 times (10s apart), curl EC2_HOST/health expecting HTTP 200
- Comment explaining the git pull approach (ensures compose file and configs are current on server)

Use the exact patterns from 04-RESEARCH.md. The workflow should be a complete, working file -- not a skeleton.

Create `docs/phase-04/guide.md` -- the concept guide. Follow the established concept-then-build pattern (like guide.md in phases 1-3). Cover:

1. **What is CI/CD?** -- Continuous Integration vs Continuous Delivery vs Continuous Deployment. The spectrum from manual to fully automated.
2. **GitHub Actions Fundamentals** -- Workflows, events/triggers, jobs, steps, runners. Use diagrams (ASCII) showing the relationship. Explain `on:` trigger types (push, pull_request, workflow_dispatch).
3. **Secrets and Environments** -- How GitHub manages sensitive values. Repository secrets vs environment secrets. Why environments are better (same names, different values, branch restrictions). Include table of secrets needed per environment (from RESEARCH.md Pattern 4).
4. **The Pipeline Architecture** -- Walk through the 3-job structure: lint-and-test -> build-images -> push-and-deploy. Explain `needs` (job dependencies), `if` conditions, and the `environment` dynamic selection. ASCII diagram of the flow.
5. **Caching in CI** -- Why caching matters (faster builds). pnpm store cache via setup-node. Docker layer caching concept.
6. **Deploy Strategy: SSH Pull** -- Why SSH + docker compose for this phase (building on Phase 2-3 manual deploy). How it will evolve to ECS in Phase 6.
7. **Health Checks** -- Why post-deploy verification matters. What happens without it (silent failures).
8. **CI/CD Tool Comparison** -- Brief table: GitHub Actions vs Jenkins vs GitLab CI vs CircleCI. Strengths/weaknesses. Why GitHub Actions for this project (native integration, free for public repos).
9. **Production Best Practices** -- OIDC instead of access keys, branch protection rules, required status checks, deployment approvals. Mark these as "things to know for real projects" (not implemented here).

The guide should be 300-500 lines, educational, with the same voice/tone as prior phase guides.
