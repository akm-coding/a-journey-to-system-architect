**04-networking-vpc/guide.md** (~120-160 lines):
- 2-3 paragraphs WHY: Why networking matters (your app is useless if nothing can reach it). Why AWS does not just give you "the internet" -- you build your own network. Why public vs private subnets exist (database should NOT be publicly accessible).
- WHAT sections:
  - VPC: Your own isolated network in AWS. CIDR block explanation with clear table: /16 = 65,536 IPs, /24 = 256 IPs. Standard pattern: VPC = 10.0.0.0/16.
  - Subnets: Carving the VPC into segments. Public (has route to internet) vs private (no direct internet route). Standard pattern: public = 10.0.1.0/24, private = 10.0.2.0/24. Availability zones.
  - Internet Gateway (IGW): Attaches to VPC, enables internet access. Only one per VPC.
  - NAT Gateway: Allows private subnet resources to reach the internet (for updates) without being reachable FROM the internet. Mention cost ($0.045/hr + data).
  - Route Tables: Rules for where traffic goes. Public subnet route table: 0.0.0.0/0 -> IGW. Private subnet: 0.0.0.0/0 -> NAT gateway (or no internet).
  - ASCII diagram: Full VPC architecture showing VPC boundary, public subnet with EC2 + IGW route, private subnet with RDS + NAT route, internet gateway at the edge. This diagram is CRITICAL -- spend effort making it clear.
  - Second ASCII diagram: Request flow from internet -> IGW -> route table -> subnet -> EC2
  - Note: "Draw this diagram yourself on paper or in Excalidraw before moving to the exercise. Drawing reinforces understanding." (per CONTEXT.md locked decision)
  - Link to AWS VPC docs, subnet docs, route table docs

**05-security-groups/guide.md** (~100-130 lines):
- 2-3 paragraphs WHY: Security groups are your firewall. Default-deny means nothing gets in unless you allow it. Misconfigured SGs are the #1 cause of "it doesn't work" in AWS.
- WHAT sections:
  - Inbound vs outbound rules. Default: all outbound allowed, all inbound denied.
  - Rule anatomy: protocol (TCP/UDP/ICMP), port range, source (CIDR or another SG)
  - STATEFUL nature: If inbound traffic is allowed, the response is automatically allowed. Contrast with Network ACLs (stateless). ASCII diagram showing stateful flow.
  - SG referencing: One SG can reference another as source. Pattern: web-sg allows HTTP from anywhere, db-sg allows PostgreSQL (5432) only from web-sg.
  - Best practices: Least privilege (only open what you need), restrict SSH to your IP (/32), never 0.0.0.0/0 on SSH in production.
  - Common patterns table: SSH (22 from your-ip/32), HTTP (80 from 0.0.0.0/0), HTTPS (443 from 0.0.0.0/0), PostgreSQL (5432 from web-sg), custom app (3000 from specific CIDR)
  - Pitfall callout: "What happens if you remove port 22? You lock yourself out. This is recoverable via AWS Console (modify the SG) but not via SSH." (from research pitfall #3)
  - Link to AWS Security Groups docs

**05-security-groups/exercise.md** (~100-120 lines):

**06-dns/guide.md** (~100-130 lines):
- 2-3 paragraphs WHY: DNS is how humans reach your app (no one memorizes IP addresses). Understanding DNS debugging (dig, nslookup) saves hours when "the site is down" is actually "DNS is wrong."
- WHAT sections:
  - How DNS resolution works: Browser -> local cache -> recursive resolver -> root nameserver -> TLD nameserver -> authoritative nameserver -> IP address. ASCII diagram of this full chain.
  - Record types:
    - A record: domain -> IPv4 address (most common)
    - CNAME record: domain -> another domain (alias)
    - MX record: mail routing (mention, don't deep dive)
    - TXT record: verification, SPF (mention)
    - NS record: nameserver delegation
  - TTL (Time to Live): How long DNS records are cached. Why low TTL during changes, higher TTL for stability. Common values: 300 (5 min) for testing, 3600 (1 hr) for production.
  - DNS propagation: Why changes are not instant. How to check propagation status.
  - Cloudflare vs Route 53: Per research recommendation, use Cloudflare free tier for this phase. Explain Route 53 is the AWS-native choice for production (alias records, health checks, weighted routing) but costs $0.50/month per hosted zone. Note the learner will use Route 53 in later phases when using Terraform.
  - Link to AWS Route 53 docs, Cloudflare DNS docs

**06-dns/exercise.md** (~100-120 lines):
- Prerequisites: A domain name (note: any registrar works, suggest Cloudflare Registrar or Namecheap for cheap .dev or .com), a running EC2 instance with Nginx from previous exercises
- Step 1: Sign up for Cloudflare (free tier) and add your domain. Update nameservers at your registrar to point to Cloudflare.
- Step 2: Wait for nameserver propagation (can take up to 24-48 hrs, usually faster). Check with `dig NS yourdomain.com`
- Step 3: Create an A record: yourdomain.com -> EC2 public IP. Set TTL to 300 (for testing). Set Cloudflare proxy to "DNS only" (grey cloud) -- we want direct connection for learning, not Cloudflare proxy.
- Step 4: Verify: `dig yourdomain.com` should return the EC2 IP. `nslookup yourdomain.com` as alternative. `curl http://yourdomain.com` should return Nginx page.
- Step 5: Create a CNAME record: www.yourdomain.com -> yourdomain.com
- Step 6: Verify: `dig www.yourdomain.com` should show CNAME + A record chain. `curl http://www.yourdomain.com` works.
- Step 7: Experiment with TTL: Change the A record IP to 1.2.3.4 (wrong). Wait TTL seconds. Check with dig. Change back. Understand caching behavior.
- Step 8: Check propagation from multiple locations: use `dig @8.8.8.8 yourdomain.com` (Google DNS), `dig @1.1.1.1 yourdomain.com` (Cloudflare DNS)
- Note about checkpoint: "You need DNS working before the SSL/TLS exercise. Your domain must resolve to your EC2 instance's public IP."
- Verification: "`curl -I http://yourdomain.com` returns HTTP 200 with Nginx headers"
- Clean up: Keep DNS records (needed for SSL exercise). Keep EC2 running.
- Rebuild challenge: "Delete all DNS records. Recreate them and verify resolution. Time yourself."

**07-ssl-tls/guide.md** (~100-140 lines):
- 2-3 paragraphs WHY: HTTPS is not optional (browsers warn on HTTP, SEO penalty, user trust). TLS encrypts traffic between browser and server. Understanding the certificate chain matters when debugging "certificate errors" in production.
- WHAT sections:
  - How TLS works (simplified): Client hello -> Server hello + certificate -> Key exchange -> Encrypted connection. ASCII diagram of TLS handshake.
  - Certificate chain: Root CA -> Intermediate CA -> Your certificate. Why browsers trust certain CAs.
  - Let's Encrypt: Free, automated, trusted CA. Certbot automates the process. 90-day certificates with auto-renewal.
  - ACM (AWS Certificate Manager): CRITICAL CALLOUT per research -- "ACM certificates CANNOT be installed directly on EC2 instances. ACM works ONLY with services that integrate with it: ALB, NLB, CloudFront, API Gateway. For standalone EC2, use Let's Encrypt." This is the #1 confusion point (research pitfall #2). ACM will be used hands-on in Phase 6 when ALB is introduced.
  - Nginx SSL configuration: How Nginx terminates TLS (listens on 443, presents certificate, decrypts traffic, forwards to app on localhost).
  - Link to ACM docs, Certbot docs, Let's Encrypt docs

**07-ssl-tls/exercise.md** (~100-130 lines):
- Prerequisites: EC2 instance with Nginx running, domain pointing to EC2 (from DNS exercise), SSH access
- Step 1: Verify prerequisite -- `curl http://yourdomain.com` returns Nginx page
- Step 2: Install Certbot on EC2:
  - Amazon Linux 2023: `sudo dnf install -y certbot python3-certbot-nginx` (note: AL2023 uses dnf, not yum)
  - Include Ubuntu alternative: `sudo apt install -y certbot python3-certbot-nginx`
- Step 3: Ensure security group allows HTTPS (port 443). If not, add the rule.
- Step 4: Run Certbot: `sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com`
  - Certbot will ask for email (for renewal notices) and agreement to ToS
  - It automatically modifies the Nginx config to serve HTTPS and redirect HTTP to HTTPS
- Step 5: Verify HTTPS works:
  - `curl -I https://yourdomain.com` -- should return 200 with valid certificate
  - Visit https://yourdomain.com in browser -- should show padlock
  - `curl -I http://yourdomain.com` -- should redirect to HTTPS (301/302)
- Step 6: Inspect the certificate: `openssl s_client -connect yourdomain.com:443 -servername yourdomain.com < /dev/null 2>/dev/null | openssl x509 -noout -dates -subject -issuer`
  - Shows: subject (your domain), issuer (Let's Encrypt), validity dates
- Step 7: Test auto-renewal: `sudo certbot renew --dry-run`
  - Explain: Certbot sets up a cron/timer to auto-renew before the 90-day expiry
- Step 8: Look at what Certbot changed in Nginx config:
  - `sudo cat /etc/nginx/nginx.conf` (or `/etc/nginx/conf.d/default.conf`)
  - Identify the ssl_certificate, ssl_certificate_key, and redirect lines
- Verification: "`curl -I https://yourdomain.com` returns HTTP/2 200 with valid cert. `sudo certbot renew --dry-run` succeeds."
- Clean up: This is the culmination exercise -- keep it running to demonstrate in the phase gate. When fully done with Phase 1: terminate EC2, release Elastic IP if allocated, Certbot renewal will simply fail harmlessly on a terminated instance.
- Rebuild challenge: "Terminate the EC2 instance. Launch a new one, install Nginx, update DNS to point to new IP, install Certbot, get a new certificate. Time yourself. This is the full stack: server + web server + DNS + SSL."
