**guide.md** -- Conceptual guide (~100-150 lines):
- 2-3 paragraphs WHY: Why understanding processes matters for deployment (your app IS a process). Why permissions exist (multi-user, security). Why env vars are critical for configuration.
- WHAT sections covering:
  - Processes: What a process is, PID, parent/child. ps aux output explained (each column). top/htop for real-time monitoring. Signals: SIGTERM (graceful) vs SIGKILL (force).
  - systemd and services: What systemd is, why it matters (starts your app on boot, restarts on crash). Unit files. systemctl commands. journalctl for logs.
  - Permissions: rwx explained (read/write/execute). Numeric notation (755, 644, 400). User/group/other. chmod and chown. sudo and why not to run everything as root.
  - Environment variables: What they are, how they work (process scope). export vs .bashrc. Common patterns (NODE_ENV, PATH, DATABASE_URL). /etc/environment for system-wide.
  - ASCII diagram: Process tree showing init -> systemd -> nginx -> worker processes

**exercise.md** -- Hands-on exercise (~100-120 lines):
- Prerequisites: Running EC2 instance from previous exercise (or launch a new one)
- Step 1: Explore running processes
  - `ps aux` and explain the output columns
  - `top` (press q to quit), note CPU/memory usage
  - `ps aux | grep sshd` to find the SSH daemon
- Step 2: Install and manage Nginx as a service
  - `sudo yum install -y nginx` (Amazon Linux 2023)
  - `sudo systemctl start nginx`
  - `sudo systemctl status nginx` -- explain the output
  - `curl localhost` from the EC2 instance to verify Nginx responds
  - `sudo systemctl stop nginx`, verify it stopped
  - `sudo systemctl enable nginx` -- explain "enable" vs "start"
  - `sudo journalctl -u nginx --no-pager -n 20` to view logs
- Step 3: Practice with permissions
  - Create a script: `echo '#!/bin/bash\necho "Hello from script"' > ~/test.sh`
  - Try to run it: `./test.sh` -- fails (no execute permission)
  - `chmod +x ~/test.sh` then `./test.sh` -- works
  - Experiment: `chmod 000 ~/test.sh`, try to cat it, then `chmod 644 ~/test.sh`
  - `ls -la ~/test.sh` after each change to see permission string
  - Create a file as root: `sudo touch /opt/root-file.txt`. Try to edit as ec2-user. Use chown to change ownership.
- Step 4: Work with environment variables
  - `echo $HOME`, `echo $PATH`, `env | head -20`
  - `export MY_VAR="hello"`, `echo $MY_VAR`
  - Open a new SSH session -- MY_VAR is gone. Explain session scope.
  - Add `export MY_VAR="persistent"` to `~/.bashrc`, `source ~/.bashrc`, verify
  - Show NODE_ENV pattern: `NODE_ENV=production node -e "console.log(process.env.NODE_ENV)"` (if node installed, or just demonstrate the concept)
- Step 5: Process control
  - Start a background process: `sleep 300 &`
  - Find it with `ps aux | grep sleep`
  - Kill it with `kill <pid>`, verify with `ps aux | grep sleep`
  - Start another: `sleep 300 &`, force kill with `kill -9 <pid>`
- Verification: "Run `systemctl status nginx` -- you should see it active. Run `ls -la ~/test.sh` -- permissions should be -rw-r--r--."
- Clean up: Stop nginx, terminate EC2 instance
- Rebuild challenge: "Launch a fresh EC2 instance. Install Nginx, configure it as a service, create a script with correct permissions, set persistent env vars. Time yourself."
