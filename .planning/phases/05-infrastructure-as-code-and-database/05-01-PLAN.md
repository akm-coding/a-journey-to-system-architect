1. **What is Infrastructure as Code** -- Define IaC, explain declarative vs imperative, why Terraform over manual AWS console clicks
2. **Terraform vs Alternatives** -- Brief comparison table: Terraform vs CloudFormation vs Pulumi vs CDK (per CONTEXT.md decision). Focus on tradeoffs, not marketing
3. **HCL Syntax Fundamentals** -- Blocks (resource, data, variable, output, locals, module), attributes, expressions, string interpolation, references (`resource_type.name.attribute`), type system (string, number, bool, list, map, object)

9. **The Terraform Lifecycle** -- `init` (providers, backend), `plan` (diff), `apply` (execute), `destroy` (teardown). The plan-then-apply safety pattern. State locking during operations
10. **Bootstrap Pattern** -- Why state backend must be provisioned separately with local state. The chicken-and-egg problem. Walk through the bootstrap directory this plan creates
11. **Cost Estimation** -- Monthly cost breakdown table for all Phase 5 resources (from RESEARCH.md cost section). Emphasize destroy+rebuild for cost control

Create the bootstrap infrastructure directory with annotated .tf files (inline comments explaining each block, same pattern as Phase 3 Dockerfiles and Phase 4 workflow YAML).

**infra/bootstrap/providers.tf:**
- `terraform` block with `required_version = ">= 1.15.0"`
- `required_providers` block: `aws` source `hashicorp/aws` version `"~> 6.0"`
- NO backend block (bootstrap uses local state by design -- this IS the state backend)
- `provider "aws"` block with `region = var.aws_region`
- Inline comment explaining why no remote backend here (chicken-and-egg)

**infra/bootstrap/variables.tf:**
- `aws_region` (string, default "ap-southeast-1", description)
- `project_name` (string, description, validation: length >= 3)
- `environment` note: bootstrap is shared, not environment-specific

Per CONTEXT.md: variables include type constraints, validation blocks, and descriptions.

**infra/bootstrap/main.tf:**
- `aws_s3_bucket` for terraform state with `lifecycle { prevent_destroy = true }` (per RESEARCH.md pitfall #2)
- `aws_s3_bucket_versioning` enabled (protect against accidental state overwrites)
- `aws_s3_bucket_server_side_encryption_configuration` with AES256
- `aws_s3_bucket_public_access_block` blocking all public access
- `aws_dynamodb_table` for state locks: PAY_PER_REQUEST billing, hash_key "LockID" (string)
- `aws_ecr_repository` for the app (per CONTEXT.md: ECR is bootstrap infrastructure, shared across environments)
- ECR lifecycle policy: keep last 10 tagged images, expire untagged after 7 days (per Phase 3 decision)
- Tags on all resources: `Project = var.project_name`, `ManagedBy = "terraform"`

Each resource block should have a comment block above it explaining what it does and why, similar to the commented Dockerfiles from Phase 3.

**infra/bootstrap/outputs.tf:**
- `state_bucket_name` -- needed for environment backend config
- `state_bucket_arn`
- `dynamodb_table_name` -- needed for environment backend config
- `ecr_repository_url` -- needed for Docker push commands
- All outputs with descriptions (per CONTEXT.md decision)
