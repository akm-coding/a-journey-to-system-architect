App structure (created by Plan 01):
```
app/
  package.json          # pnpm dependencies
  ecosystem.config.js   # PM2 config: "ecommerce-api", port 3000, fork mode
  dist/                 # Vite build output (React static files)
  dist/server/index.js  # Compiled Express server
  src/server/           # Express API source
  src/client/           # React source
```

Key endpoints:
- GET /api/health -> { status: "ok", timestamp }
- GET /api/products -> product list
- POST /api/orders -> create order

Nginx serves:
- / -> React static files from app/dist/
- /api/* -> proxy to PM2/Node on port 3000

- VPC with public/private subnets (from 04-networking-vpc)
- Security groups with least-privilege (from 05-security-groups)
- DNS pointing to EC2 (from 06-dns)
- Certbot/Let's Encrypt for HTTPS (from 07-ssl-tls)

    Create docs/phase-02/01-ec2-deploy/runbook.md as a comprehensive step-by-step deployment guide. Follow the CONTEXT.md decision: "hands-on runbook style with actual commands, explain the why alongside each command, verification checkpoints after each major step, include common gotchas inline."

    Structure the runbook in the incremental layered order from CONTEXT.md:

    **Section 1: Launch EC2 and Initial Setup**
    - Create VPC using "VPC and more" wizard (2 public + 2 private subnets across 2 AZs) -- reference Phase 1 VPC knowledge, explain why 2 AZs (RDS requirement from research pitfall #1)
    - Create EC2 security group: inbound SSH (22) from your IP, HTTP (80), HTTPS (443)
    - Launch t2.micro with Amazon Linux 2023, attach Elastic IP
    - SSH in, install Node.js 20 (`sudo dnf install -y nodejs20`), handle the `alternatives` issue from research pitfall #3
    - Install pnpm globally
    - Verification checkpoint: `node --version` and `pnpm --version` work after SSH reconnect

    **Section 2: Deploy and Run Node API Directly**
    - Clone the repo on EC2 (or git init + push -- use `git clone` per research recommendation)
    - cd into app/, run `pnpm install`
    - Set up environment variables (DATABASE_URL will come later, use a placeholder or skip DB-dependent routes for now)
    - Run the API directly: `node dist/server/index.js` (after building)
    - Verification checkpoint: `curl http://localhost:3000/api/health` returns 200
    - Explain: why this works but is not production-ready (no auto-restart, blocks terminal, runs as foreground process)

    **Section 3: Install and Configure PM2**
    - Install PM2 globally: `sudo npm install -g pm2`
    - Copy ecosystem.config.js, start with `pm2 start ecosystem.config.js`
    - Essential commands: `pm2 status`, `pm2 logs ecommerce-api`, `pm2 restart ecommerce-api`
    - Set up startup persistence: `pm2 startup` (run the printed sudo command) + `pm2 save`
    - Explain each PM2 command and WHY startup/save matter (research pitfall #6)
    - Verification checkpoint: `pm2 status` shows "online", `curl localhost:3000/api/health` works

    **Section 4: Install and Configure Nginx as Reverse Proxy**
    - Install Nginx: `sudo dnf install -y nginx && sudo systemctl enable nginx && sudo systemctl start nginx`
    - Explain what a reverse proxy does and why (port 80/443 handling, static file serving, SSL termination)
    - Create /etc/nginx/conf.d/ecommerce.conf with the dual-purpose config from research:
      - `root` pointing to /home/ec2-user/app/dist (React static files)
      - `location /` with `try_files $uri $uri/ /index.html` (React Router support)
      - `location /api/` proxying to http://127.0.0.1:3000 with all proxy headers
    - Remove or disable the default nginx config if it conflicts
    - Always run `sudo nginx -t` before reload (research pitfall #4)
    - `sudo systemctl reload nginx`
    - Explain each directive in the config (what it does, why it matters)
    - Verification checkpoint: From your local machine, `curl http://<elastic-ip>/api/health` returns 200

    **Section 5: Build and Serve React Frontend**
    - On EC2, build the React frontend: `cd app && pnpm run build:client`
    - Verify build output exists in app/dist/ (index.html, assets/)
    - Nginx already configured to serve from this directory
    - Verification checkpoint: Visit http://<elastic-ip> in browser, see the product list page
    - Troubleshooting: blank page = check browser console, 404 on API = check relative URLs (research pitfall #5)

    **Section 6: Set Up HTTPS with Let's Encrypt**
    - Prerequisites: DNS must already point to the Elastic IP (reference Phase 1 DNS exercise)
    - Verify DNS: `dig yourdomain.com` should return the Elastic IP
    - Install Certbot via pip per research (AL2023 has no certbot package):
      ```
      sudo python3 -m venv /opt/certbot/
      sudo /opt/certbot/bin/pip install certbot certbot-nginx
      sudo ln -s /opt/certbot/bin/certbot /usr/bin/certbot
      ```
    - Run `sudo certbot --nginx -d yourdomain.com`
    - Explain what Certbot does (domain validation via HTTP-01, modifies Nginx config, installs cert)
    - Set up auto-renewal cron per research
    - Verify: `sudo certbot renew --dry-run`
    - Research pitfall #8: Certbot fails if DNS not pointing to EC2 -- diagnose with `dig`
    - Verification checkpoint: Visit https://yourdomain.com in browser, see green lock

    **Section 7: Rebuild Shortcut**
    - Quick steps to re-deploy after code changes:
      ```bash
      cd ~/app && git pull && pnpm install && pnpm run build:client && pnpm run build:server && pm2 restart ecommerce-api
      ```
    - When to rebuild from scratch vs incremental update

    - Include [CHECKPOINT] markers with exact curl/browser verification steps
    - Include [GOTCHA] callouts for common mistakes from research pitfalls
    - Include [WHY] callouts explaining the reasoning behind non-obvious steps
    - Keep the tone practical and direct -- a runbook, not a textbook

    - Common troubleshooting: "API not responding" -> check pm2 status/logs, "Nginx won't start" -> nginx -t, "HTTPS fails" -> check DNS with dig
