    Create docs/phase-02/02-rds-database/runbook.md as a step-by-step guide. Same style as the EC2 runbook: commands with "why" explanations, verification checkpoints, and gotcha callouts.

    **Section 1: Create RDS Security Group**
    - Create a new security group for RDS (name: ecommerce-rds-sg) in the same VPC as EC2
    - Inbound rule: PostgreSQL (port 5432), source = EC2 security group ID (NOT an IP address)
    - Explain SG referencing pattern: "This means only instances in the EC2 security group can reach the database. If you add a second EC2 instance to that SG, it automatically gets database access too."
    - Reference Phase 1 security groups exercise where this pattern was introduced conceptually
    - [GOTCHA] Research pitfall #2: Using an IP address or 0.0.0.0/0 instead of SG reference -- explain why this is wrong and dangerous
    - Verification checkpoint: Describe the SG rule in the console, confirm source shows "sg-xxxxx" not an IP

    **Section 2: Create DB Subnet Group**
    - Explain: RDS requires a DB subnet group spanning at least 2 AZs (even for single-AZ deployment)
    - [GOTCHA] Research pitfall #1: This is the most common RDS creation error. AWS error message: "DB Subnet Group doesn't meet availability zone coverage requirement."
    - If VPC was created with "VPC and more" wizard in the EC2 runbook, the 2 private subnets already exist. Identify them.
    - Create DB subnet group using AWS CLI or console, selecting ONLY the 2 private subnets
    - Explain: private subnets have no internet gateway route, so RDS is not accessible from the internet
    - Verification checkpoint: `aws rds describe-db-subnet-groups` shows the group with 2 subnets in different AZs

    **Section 3: Launch RDS PostgreSQL Instance**
    - Create RDS instance with these EXACT settings (free tier -- research pitfall #7):
      - Engine: PostgreSQL 16.x
      - Instance class: db.t3.micro (free tier eligible)
      - Storage: 20 GB gp2 (free tier limit)
      - Multi-AZ: NO (not free tier)
      - Public access: NO (private subnet only)
      - DB subnet group: the one just created
      - Security group: ecommerce-rds-sg
      - Master username: postgres (or ecommerce_admin)
      - Master password: set a strong password, save it securely
      - Initial database name: ecommerce
      - Performance Insights: OFF (not free tier)
      - Automated backups: 7 days retention (free)
      - Encryption: default (free)
    - Show both Console and CLI approaches
    - Explain: RDS takes 5-10 minutes to create. Wait for status "Available"
    - [GOTCHA] Research pitfall #7: Multi-AZ, Performance Insights, and storage >20GB cause charges
    - Verification checkpoint: `aws rds describe-db-instances --db-instance-identifier ecommerce-db` shows "available"

    **Section 4: Connect Express API to RDS**
    - Get the RDS endpoint from the console or CLI: `aws rds describe-db-instances --query 'DBInstances[0].Endpoint.Address'`
    - Construct DATABASE_URL: `postgresql://username:password@rds-endpoint:5432/ecommerce`
    - SSH into EC2 and test connectivity first: `psql -h <rds-endpoint> -U postgres -d ecommerce` (install psql if needed: `sudo dnf install -y postgresql16`)
    - [GOTCHA] Research pitfall #2: If connection times out, check: (1) RDS SG references EC2 SG, (2) RDS is in the private subnets, (3) EC2 is in the same VPC
    - Set DATABASE_URL as environment variable on EC2:
      - For PM2: add to ecosystem.config.js env section or use a .env file with dotenv
      - Explain: NEVER commit .env with real credentials. On EC2, set in the ecosystem file or export in ~/.bashrc
    - Verification checkpoint: `curl http://localhost:3000/api/health` still returns 200 (API starts with DB connection)

    **Section 5: Run Migrations and Seed Data**
    - On EC2, run schema push: `cd ~/app && DATABASE_URL="..." pnpm run db:push`
    - Explain: `drizzle-kit push` applies the schema.ts definitions to the database (creates tables)
    - Run seed script: `DATABASE_URL="..." pnpm run seed`
    - Verify data: `psql -h <rds-endpoint> -U postgres -d ecommerce -c "SELECT count(*) FROM products;"`
    - Restart PM2 to pick up DATABASE_URL: `pm2 restart ecommerce-api`
    - Verification checkpoint: `curl http://<elastic-ip>/api/products` returns JSON array of seeded products
    - Final verification: Visit the app in the browser, see products loaded from the real database

    **Section 6: Understanding the Full Stack**
    - Diagram recap: Browser -> Nginx (port 443) -> PM2/Node (port 3000) -> RDS (port 5432)
    - Explain each hop: what happens at each layer, what would break if removed
    - Security recap: Nginx handles SSL, EC2 SG limits inbound, RDS SG limits to EC2 only
    - Cost recap: what you're paying for (EC2 750hrs free, RDS 750hrs free, Elastic IP free when attached)

    Create docs/phase-02/02-rds-database/cheatsheet.md:
    - RDS CLI commands: create, describe, delete, wait
    - Security group commands for RDS
    - psql connection command
    - Drizzle commands: db:push, seed
    - DATABASE_URL format reference
    - Troubleshooting: connection timeout -> SG rules, auth failed -> credentials, table not found -> run db:push

    Update scripts/teardown-phase2.sh (replacing the generic teardown-checklist.sh content for Phase 2):
    - Shebang: #!/bin/bash, set -e
    - Print a banner: "Phase 2 Teardown -- Destroy All AWS Resources"
    - Print a warning: "This will delete all Phase 2 resources. Resources must be deleted in dependency order."
    - Ask for confirmation (read -p "Are you sure? (yes/no): ")
    - Prompt for required IDs at the start: EC2 instance ID, Elastic IP allocation ID, RDS instance identifier, security group IDs, VPC ID, DB subnet group name
    - Delete in correct dependency order (per research teardown script):
      1. Stop PM2 and clean up on EC2 (optional if terminating)
      2. Delete RDS instance (skip final snapshot to save cost): `aws rds delete-db-instance --db-instance-identifier $RDS_ID --skip-final-snapshot`
      3. Wait for RDS deletion: `aws rds wait db-instance-deleted --db-instance-identifier $RDS_ID`
      4. Delete DB subnet group: `aws rds delete-db-subnet-group --db-subnet-group-name $DB_SUBNET_GROUP`
      5. Terminate EC2 instance: `aws ec2 terminate-instances --instance-ids $EC2_ID`
      6. Wait for EC2 termination
      7. Release Elastic IP: `aws ec2 release-address --allocation-id $EIP_ALLOC_ID`
      8. Delete security groups (RDS SG first, then EC2 SG -- order matters if SGs reference each other)
      9. Delete VPC: `aws ec2 delete-vpc --vpc-id $VPC_ID` (this cleans up subnets, route tables, IGW)
    - Each step prints what it's doing and waits for completion before proceeding
    - Include comments explaining WHY the order matters (dependency chains)
    - Print final summary: "All Phase 2 resources destroyed. Monthly charges will stop."
